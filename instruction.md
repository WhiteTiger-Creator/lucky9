Evaluate the Meridian-2 network transport observable for a directed flow network and report the sustained flux.

Meridian-2 is a directed flow network from the statistical physics of transport: flux enters at a single source and is carried downstream along transport channels that contend for shared sites. The task evaluates the sustained-flux observable, the routed channel set that realizes it, several derived aggregates, and integrity checksums.

How the observable is evaluated is **not** in the output contract and you must reconstruct it. Link conditioning and validity bounds, the channel span bound, the sustained-flux objective itself, the deterministic tie-broken routed channel set, the residual packing, the per-hop efficiency aggregates, the site-conditioning and damping rules, and the dispatch admission, class and ordering layers were settled by the Meridian-2 model review board, and `/app/incident/flux_calibration_log.md` is the authoritative record of every one of them. Most of the log is routine noise, and the February draft parameterizations were partly reversed later; several were revised again in the 2026-05 calibration, so trace each rule to its final decision — where a draft or interim conflicts with a later decision, the later decision governs. The natural first guesses at the objective are drafts the board explicitly considered and rejected; read the log rather than assume.

`/app/docs/model_spec.md` is the output contract only: it fixes the network input shape, the exact `result.json` key set, and the byte-level checksum serialization — not how the values are derived.

Site conditioning is read from `/app/data/site_conditioning.json` at that fixed absolute path; `--input` selects the network only and never relocates the conditioning file. Two output-vocabulary and type facts the verifier matches literally, so that reconstructing a stage correctly is never defeated by guessing its spelling or container: a dispatched channel's class is exactly one of the lowercase strings `primary`, `secondary`, or `tertiary`, and `class_counts` always enumerates all three in that order, emitting `0` for a class with no dispatched channels; `dispatch_order` is a JSON array in the log's dispatch ordering, which is not ascending order — `dispatched_endpoints` is the sorted one. Which class a channel earns, and the ordering itself, are governed by the calibration log.

Given the network in `/app/data/network.json`, evaluate the transport and write `result.json` with the keys the spec lists. Carry out the computation in `/app/flux.py`, reading `--input` (default `/app/data/network.json`) and writing under `--output-dir` (default `/app/output`), and run it so `/app/output/result.json` is in place. Derive every value from the input and the governing calibration decisions; don't read or import anything from `/tests` or `/solution`.
