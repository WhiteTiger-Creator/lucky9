You are the incident responder on the Warden endpoint process-execution containment platform during an active breach. The automated control that isolates hostile process executions — deciding which executions on each host are contained and escalated to responders — has been tampered with and is now letting malicious processes run on the fleet instead of containing them. Restoring correct containment before the intruder pivots further is the security objective; every host stays exposed for as long as the isolation decisions are wrong.

The tampered control lives at `/app/workflow/export_report.py`. It reads the raw execution-telemetry stream off the endpoint sensors and must once again decide containment correctly, leaving three records the responders act on: the incident summary `summary.json`, the per-host exposure map `host_matrix.json`, and the isolation queue `contained.jsonl`. It always reads the sandbox and audit control windows from their fixed absolute path `/app/data/exec_policies.json`, and the host service-dependency graph from its own fixed absolute path `/app/data/host_dependency_edges.json`: `--input` reparameterizes only the telemetry stream, so neither of those files ever becomes relative to the input or output directory, and both are read from those same absolute paths even when the telemetry is loaded from elsewhere. No triage tooling survived the tampering, so you must **create** `/app/exec_audit.py` — it does not exist yet — exposing two actions. `diagnose --dossier PATH --report PATH` reads the review log at the `--dossier` path and **writes** its report of the six known containment defects — with the forensic evidence for each — to the `--report` path, which is an output file it creates (not an input to read). `repair --output-dir PATH` restores the control from its frozen pre-incident snapshot, re-runs containment, and records the containment diagnosis (`diagnosis.json`) and the repair audit (`repair_audit.json`) — which is incomplete with the hash and token fields alone: it must also carry `schema_version` (the literal `warden-repair-v2`), `defects_repaired` (the sorted list of repaired defect id strings, not a count) and `artifacts` (the sorted list of artifact file names written). The defect ids are the canonical ones in the contract's `known_defects` — `admission_filter`, `wrong_entry_field`, `queue_ordering`, `run_class_normalization`, `dedupe_exec` and `killed_filter` — never ids of your own — beside the three records, defaulting to `/app/output`.

Every containment threshold — how execution classes are ranked, when a hostile execution is isolated, how repeated exposure accumulates against a host, how far a compromised host's lateral reach is scored across the service-dependency graph, and which executions cross into the responder queue — was ruled on by the execution-review board while the incident was live, and those rulings survive only in the review dossier at `/app/incident/exec_review_dossier.md`, where a later dated ruling supersedes an earlier one. Most of that log is routine incident chatter, so each threshold must be traced to its final ruling rather than the first mention. The exact record and digest formats the responders' tooling validates against are pinned in `/app/docs/report_spec.json`. The frozen pre-incident snapshot at `/app/workflow/.export_report.original` is read-only forensic evidence and must not be altered. Containment is judged on behaviour: it must hold against an alternate execution-telemetry capture it has never seen, reach identical decisions on repeated runs, and derive every containment call from the telemetry and the board's rulings alone.
