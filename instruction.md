Evaluate the Meridian-2 network transport observable for a directed flow network and report the sustained flux.

Meridian-2 is a directed flow network from the statistical physics of transport: flux enters at a single source and is carried downstream along transport channels that contend for shared sites. The task evaluates the sustained-flux observable, the routed channel set that realizes it, several derived aggregates, and integrity checksums.

The evaluation rules are not part of the output contract. `/app/incident/flux_calibration_log.md` is the authoritative source for all of them: link conditioning and validity bounds, the channel span bound, the sustained-flux objective, the deterministic tie-broken routed channel set, the residual packing, the per-hop efficiency aggregates, the site-conditioning and damping rules, and the dispatch admission, class and ordering layers.

The log records these rules as dated review entries, and more than one entry addresses each rule: February draft parameterizations and April interim decisions were revised by the 2026-05 calibration cycle. For every rule the governing value is the one stated in its latest dated decision. Earlier drafts and interim entries are superseded and carry no force, including where a superseded entry states an otherwise plausible alternative.

`/app/docs/model_spec.md` is the output contract only: it fixes the network input shape, the exact `result.json` key set, and the byte-level checksum serialization — not how the values are derived.

Site conditioning is read from `/app/data/site_conditioning.json` and the transport policies from `/app/data/transport_policies.json`, both at those fixed absolute paths; `--input` selects the network only and never relocates either file. The dispatch and ledger thresholds are not global constants: they are resolved per channel from the policy file by the channel's ingress hub, through a layered default-then-override inheritance whose tiers, key and serialization the calibration log specifies. Two output-vocabulary and type facts the verifier matches literally, so that reconstructing a stage correctly is never defeated by guessing its spelling or container: a dispatched channel's class is exactly one of the lowercase strings `primary`, `secondary`, or `tertiary`, and `class_counts` always enumerates all three in that order, emitting `0` for a class with no dispatched channels; `dispatch_order` is a JSON array in the log's dispatch ordering, which is not ascending order — `dispatched_endpoints` is the sorted one. Which class a channel earns, and the ordering itself, are governed by the calibration log.

Given the network in `/app/data/network.json`, evaluate the transport and write `result.json` with the keys the spec lists. Carry out the computation in `/app/flux.py`, reading `--input` (default `/app/data/network.json`) and writing under `--output-dir` (default `/app/output`), and run it so `/app/output/result.json` is in place. Derive every value from the input and the governing calibration decisions; don't read or import anything from `/tests` or `/solution`.
